When a creature slides without moving

the anomaly

Two boxes, one joint, no visible gait

The phenotype is tiny. Bodies are 0.21×0.33×0.80 m and 0.81×0.38×0.50 m. They're connected by a single revolute joint with limits [−0.82, +1.06] rad. On the fitness ledger this creature scored well on SwimmingSpeed in a Land environment. Visually, it glides across the checkerboard floor like a hockey puck — same pose, same angle, just… moving.

⚠ What we expect

Karl Sims–style locomotion requires cyclic actuation. Limbs bend, push off the ground, return. The root body should advance because some segment is periodically pressing against the floor.

🧩 What we see

A static pose sliding at up to 6 m/s. No visible wiggling, no gait cycle. The joint appears frozen. This can't be normal friction-driven walking — something else is producing horizontal force.

hypothesis list Candidates before looking at data: (a) the fitness metric rewards something that isn't locomotion; (b) a joint-constraint artefact in the PGS solver produces phantom forces; (c) micro-oscillations below the frame sample rate pump the body forward; (d) ground friction + a biased contact geometry creates drift.
step 1 · reach for the CLI

The debug binary already exists

karl-sims-debug loads a genome straight from the SQLite file, runs the identical physics the fitness evaluator runs, and emits a per-frame JSON trace. You don't need to boot the server or the viewer to investigate a creature — the deterministic physics means the CLI produces bit-identical output.

bash — reproduce creature 794863 locally
# 1. Build the debug bin once (release mode — fitness is release-timed) $ cargo build --release --bin karl-sims-debug # 2. Run the simulation, dump a trace $ ./target/release/karl-sims-debug \ --evolution 45 --creature 794863 \ --verbose --output /tmp/trace-794863.json # stderr: evaluates params, simulates 600 frames (10 s @ 60 Hz), # prints root position every second, summarizes drift + NaN frames. # stdout: — (the JSON goes to the --output path)
1Reads genotypes.genome_bytes by ID, deserializes via bincode into a GenomeGraph
2Reads evolutions.config_json, parses EvolutionParams (environment, gravity, sim_duration)
3Applies identical environment setup as evaluate_fitness: gravity, ground plane, root at y=2
4Steps Rapier at 60 Hz for sim_duration seconds, collecting per-frame body transforms
5Writes JSON: one frame per entry, each frame has all body positions + quaternions + half-extents
step 2 · read the trace

The JSON schema is flat and small

601 frames, two bodies each, ~18 KB. Small enough that a Python one-liner can scan for anomalies. The key fields are px/py/pz (world position) and qw/qx/qy/qz (orientation) — those are enough to see what the body is doing.

python — anomaly probe
import json t = json.load(open('/tmp/trace-794863.json')) # Is the joint actually moving? qxs = [f['bodies'][1]['qx'] for f in t['frames']] for sec in range(10): w = qxs[sec*60:(sec+1)*60] print(f't={sec}-{sec+1}s: range={max(w)-min(w):.4f}') # t=0-1s: range=0.5414 ← falling + bending # t=1-2s: range=0.0056 ← frozen # t=2-3s: range=0.0000 ← frozen # t=3-4s: range=0.0000 ← frozen # t=4-5s: range=0.0000 ← still frozen # t=5-6s: range=0.0000 ← still frozen # t=8-9s: range=0.9390 ← finally flips
what this tells us The child body's orientation quaternion (specifically qx, which corresponds to rotation about the revolute axis) has zero variation across five full seconds. If the joint were oscillating — even at 100 Hz below our sample rate — we'd see jitter in the noise floor. We don't. The joint is mechanically locked.

step 4 · inspect the brain

The actuator is a very slow sine wave

Deserialize the genome and print its brain nodes. Node 1 has one neuron, one effector. The neuron is an OscillateWave with tiny frequency and a small weight — so tiny that over the 10-second simulation, the sine barely completes one cycle.

genome · node 1 brain
OscillateWave { inputs: [ (Constant(1.0237), weight 0.6319), // freq (Constant(4.5407), weight 0.1223), // phase ] } // effective freq = 1.0237 · 0.6319 = 0.647 rad/s // period = 2π/0.647 ≈ 9.71 s ← barely one cycle in 10 s sim Effector { input: Neuron(0), weight: 0.2017 // tiny · output ∈ [-0.20, +0.20] }
sub-signal Output of the effector over 10 s: sin(t·0.647 + 0.555) · 0.2017. That's a slowly varying signal between −0.20 and +0.20, never zero-crossing during the fast SLIDE phase — the torque direction never flips, so the joint stays pressed against the same limit.

effector output · 10 s window

slow sine — stays positive-signed for ~7 s, flips once near t=8
the deception The creature does have a periodic brain — but evolution tuned the frequency so low that within the fitness-evaluation window, the sine effectively behaves as a constant. What looks like an oscillator on paper acts as a DC bias in practice.
step 5 · the mechanism

How a static torque pushes the body sideways

A pure torque couple on a free body produces rotation, not translation. But once the joint hits its limit and the child's bottom face contacts the ground, the torque's reaction forces get channeled through the contact point — and a horizontal component leaks out through the ground-friction coupling.

● What evolution found

A brain that keeps the joint pinned on one side. The small sine weight (0.2) and slow frequency (~0.6 rad/s) makes this effectively a DC torque within the fitness window.

● Why it translates

The joint limit converts a rotational command into static pressure against the parent. That structural load passes through the contact point and, because of the parent's tilt, has a non-vertical component.

● Why it's an exploit

This isn't locomotion — it's a ground-contact-meets-joint-limit solver quirk that produces drift without doing the mechanical work of walking. Fitness rewards displacement, so evolution finds it.

why it stops at t=8 s Look at the oscillator output chart. At ~t=7.8 s the sine finally crosses zero and the torque direction flips. The joint releases from its negative limit and swings across to its positive limit. But the new configuration's horizontal-force component points opposite the drift — net force drops to near zero, and the creature comes to rest. The trace confirms: rz only changes by 7 cm from t=8 s to t=10 s.
step 6 · what to do about it

Root cause & candidate fixes

Yes — we found the root cause. The fitness function rewards root displacement, and the physics allows a constant-torque-against-a-joint-limit strategy to produce displacement without gait. Below are four candidate fixes, from most targeted (selection pressure) to most invasive (physics changes, which CLAUDE.md says we must avoid without approval).

fix #1 · fitness · easiest · highest leverage

Require joint-angle variance

Multiply the speed score by a motion coefficient: clamp(mean_joint_stddev / threshold, 0, 1). A creature whose joints never change angle gets score=0. Legitimate walkers are unaffected because they already cycle their joints. No paper divergence — Sims-1994 creatures all exhibit cyclic gaits, so this just formalizes what's already implied.

fix #2 · brain · moderate

Clamp slow oscillator frequencies

Currently OscillateWave accepts any frequency input. A creature with a 0.1 Hz sine wave is functionally a DC brain within a 10 s eval. Enforce a minimum effective frequency (e.g., ≥ 1 Hz) on OscillateWave, or add a mutation bias that pulls freq toward the middle of a biologically plausible range (1–5 Hz).

fix #3 · brain · surgical

Zero torque when joint is at limit

In apply_effectors, skip writing torque for a DOF when joint.angles[dof] is already beyond the limit in the direction the torque would push. This removes the "lean forever" strategy at the source — a static torque against a limit stops being free. Doesn't affect cyclic actuators, which oscillate through the midrange.

fix #4 · physics · REQUIRES APPROVAL

CLAUDE.md: "Do NOT modify physics code"

Increase linear damping on land

Current lin_damp=0.3 gives a 6 m/s terminal velocity under a 0.2 N·m leaked horizontal force. Raising to 1.5 would cut terminal drift to ~1.2 m/s, drastically reducing fitness reward for the exploit. But this is in rapier_world.rs — surface to user before touching.

recommendation Start with Fix #1 (joint-variance fitness coefficient). It's the cheapest to implement, safest to revert, and directly targets the selection pressure that rewards static-pose sliders. If post-fix evolution still produces low-variance winners, layer Fix #3 on top. Keep Fix #4 in reserve pending approval — physics changes need TDD tests per the project rules.
